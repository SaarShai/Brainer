# Per-skill test-method design — 2026-06-13

**Goal:** 3 new, reliable ways to test *each* skill that (1) **quantify its gains** in real
units, (2) **reveal how it operates** so we can learn from it, and (3) **surface
efficiency/reliability improvements**.

**Method:** one agent per skill read the skill's `SKILL.md` + `tools/` + existing coverage
(`run_all_tests.sh`, `test_*.py`, `eval/`) and proposed exactly 3 methods — one per intent.
16 skills × 3 = **48 methods**. Full per-skill detail is below the marker; this top section
is the synthesis. Distribution: 27 deterministic · 15 seeded-stochastic · 6 LLM-judged;
8 low / 32 med / 8 high effort.

---

## The core finding: 48 methods → 4 reusable harnesses

The same three shapes recur across almost every skill. Rather than build 48 bespoke tests,
build **4 parameterized harnesses + a small per-skill config**. That is the efficient path
to "measure every skill's gains and learn to improve it."

### H1 · Ablation / attribution harness  *(covers the OPS method of ~14 skills)*
Zero-or-stub each internal rule/feature one at a time, re-run on a fixed fixture set, and
record how many verdicts/outputs **flip** → a per-rule importance table.
- **Skills:** cache-lint (4 Rule-2 mechanisms), write-gate (scoring features + why-gate),
  wiki-memory (9 rank factors), compliance-canary (probe kinds), verify-before-completion
  (gate components), prompt-triage (guards, leave-one-out), output-filter (dedupe/collapse/ANSI),
  lean-execution (9 protocol rules), caveman-ultra (rules), skill-pulse (discover/sort/cap),
  semantic-diff (skip_parent/stub-truncation), wiki-refresh (signal-ablation), think (methods),
  plan-first (structure axes), context-keeper (cap/dedup).
- **Shared output:** `flips_per_rule` (count) + marginal contribution (% of total effect).
- **Answers "learn how it operates":** which bullets are load-bearing vs dead prose — directly
  feeds the catalog-tax trimming the suite already optimizes for.
- **Effort:** med · mostly **deterministic** (the tool skills); LLM-judged for the prose skills.

### H2 · Fuzz + perf/ReDoS battery  *(covers the STRESS method of ~10 skills)*
Property-based fuzz (no crash, invariants hold) + an adversarial **perf benchmark with a hard
time budget** (the class of bug we just shipped twice: cache-lint typography O(n²), PATH_RE ReDoS).
- **Skills:** cache-lint, compliance-canary, context-keeper, output-filter, semantic-diff,
  skill-pulse, wiki-memory, verify-before-completion, write-gate, prompt-triage.
- **Shared output:** `crash_count` (target 0), `invariant_violations` (target 0), `p99 / worst-case
  ms` vs a committed budget, throughput (MB/s or µs/event).
- **Already have the chassis:** `eval/sims/` (now wired into `run_all_tests.sh`). Each skill becomes
  one more `eval/sims/<skill>_fuzz.py` entry with a `dos_under_Xs` assertion — exactly what caught
  the cache-lint DoS this session.
- **Effort:** low–med · **deterministic/seeded**.

### H3 · Gain-vs-baseline differential runner  *(covers the GAIN method of the tool skills)*
Run the skill vs an explicit no-skill/naive baseline on a labeled corpus; emit the benefit as a
**number + unit**. Most are deterministic token/cost/accuracy diffs.
- **Skills (deterministic):** output-filter (% tokens cut, 0% signal loss), semantic-diff (break-even
  read count R*, net tokens), skill-pulse (tok/1000-turns vs full re-inject), write-gate (% memory
  noise cut), wiki-memory (% context-budget saved), prompt-triage (net cents/100 prompts),
  cache-lint (the unmeasured "≥30% cache-hit uplift" via a static pricing model), context-keeper
  (extraction precision/F1).
- **Shared output:** one headline `gain` per skill, each in its own honest unit (tokens, %, cents,
  reads-to-break-even) **with the baseline named** — this is what lets gains be stated in multiple
  units regardless of plan/usage.
- **Effort:** low–med · **deterministic** (re-uses existing `eval/runner_*` baselines).

### H4 · Behavioral-eval harness (with/without arms + judge)  *(the prose skills' GAIN+OPS)*
The 6–7 prose skills ship no code, so their gains are behavioral. One harness: same fixed prompts,
fixed model+temp+seeds, run with vs without the skill body, score with a rubric (regex for
hard signals, LLM-judge for soft ones) + report mean ± stddev across seeds.
- **Skills:** caveman-ultra, think, lean-execution, plan-first-execute, verify-before-completion
  (prose side), index-first (verb-selection), wiki-refresh (5-outcome accuracy).
- **Shared output:** with-vs-without delta (e.g. think: sycophancy −35pp; caveman: net tok/turn;
  plan-first: bypass-F1; index-first: verb accuracy).
- **Effort:** med–high · **llm-judged-bounded** (the only stochastic harness — needs seed + variance
  discipline; this is the reliability-risk tier).

Build order is **H2 → H1 → H3 → H4**: lock perf first (active bug class), then attribution
(cheap + high learning), then gains (deterministic), then the stochastic behavioral tier last.

## Gaps / reliability risks

- **The 6 LLM-judged methods are the only non-reproducible tier.** They need fixed model+temp,
  ≥3 seeds, a pinned judge rubric, and reported variance — otherwise "gain" numbers drift run-to-run.
  Treat H4 as a measurement *aid*, not a CI gate.
- **Hardest to test reliably:** `think` and `lean-execution` (diffuse, judgment-heavy effects; the
  ablation can only show *named-but-not-done* "theater" rates, not true causal lift) and `wiki-refresh`
  (5-outcome accuracy needs a hand-labeled drift corpus that itself ages).
- **Watch for weak determinism claims:** several STRESS methods call graphify/ollama internals
  "seeded-stochastic" — they need a 3× variance report, not a single number.
- **Don't over-build:** 48 bespoke tests would itself be catalog-tax bloat. The 4-harness
  consolidation is the lean path; per-skill specifics become config rows, not new files.

## How this layer pays off

Today the suite proves *correctness* (40/40) but states most *gains* as estimates. These four
harnesses turn each skill into a measured object: **H3** gives every tool-skill a headline gain in
its own honest unit (so gains can be quoted in tokens, %, cents, or reads regardless of the user's
plan); **H1** shows which internal rules actually earn the savings (so trimming is evidence-led, not
guessed); **H2** turns "fast enough" into a committed budget that fails CI on a regression (the bug
class that bit us twice today); **H4** finally puts numbers on the prose skills. Together they make
"how efficient/reliable is skill X, and what should change" a query you *run*, not estimate.

## cache-lint  ·  has_tools=True

**How it operates:** cache-lint is a deterministic static linter (skills/cache-lint/tools/cache_lint.py, ~700 lines, stdlib-only) that discovers a project's prompt-cache prefix surface (CLAUDE.md/AGENTS.md/GEMINI.md, .claude/settings*.json, hooks JSON, skills/*/SKILL.md) and runs six rule-checkers, emitting typed Finding dataclasses with severity OK/WARN/FAIL and process exit codes 0/1/2/3. Single-run rules (2 dynamic content, 4 model switching, 5 sizing, 6 fork safety) are pure functions of file contents; stateful rules (1 ordering, 3 tool-description stability) diff a SHA256 fingerprint (.cache-lint-fingerprint.json) against the prior run. Rule 2 is the most mechanism-rich: it pre-computes inline-code spans (_inline_code_spans) and fence positions once per file, then for each of 9 DYNAMIC_PATTERNS applies a typography-skip (backticked `$(...)` ignored via _pos_in_spans), a fence-parity downgrade (FAIL->WARN inside ``` blocks via _inside_fence_fast), and a per-label cap of 5 (MAX_FINDINGS_PER_RULE_PER_FILE). Rule 6 follows hook commands into invoked scripts (SCRIPT_INVOKE_RE) and greps them for prefix-file writes (INLINE_WRITE_RE + PREFIX_FILE_RE). Comments cite a deliberate O(n^2)->O(n log n) optimization for adversarial single-line inputs.

**Already tested:** skills/cache-lint/tools/test_cache_lint.py — 20 unit/regression tests (wired into scripts/run_all_tests.sh:47), one per rule plus regressions for read-only hooks, script-following, nested discovery, node_modules skipping, sizing thresholds, non-dict settings, inline-triple-backtick fence parity, per-label cap correctness, block-scalar description drift. eval/exp10_cache_lint/run_cache_lint_eval.py — 18-case labeled fixture-DIR corpus measuring in-distribution detection precision/recall/F1/false-alarm for rules 2/4/5/6 (achieved F1=1.0, recall=1.0, FP=0.0). eval/exp13_cache_lint_ood/run.py — out-of-distribution: Part 1 = 13 new-surface-form labeled fixtures (backtick, ${VAR}, $VAR, jinja, bedrock ARNs, sed -i) for recall/precision; Part 2 = runs against the real Brainer repo and adjudicates FAILs. GAPS: (a) the EVAL.md target '>=30% cache-hit/$ uplift' is explicitly documented as UNMEASURED; (b) no test reveals WHICH internal mechanism (typography-skip/fence-parity/cap/model-key) drives a given verdict; (c) the O(n^2) DoS hazard the code comments claim to guard ('dos_under_2s case') has NO committed perf/timing regression test — I confirmed current pathological 5000x$(date) runs in ~6ms but nothing locks that in.

### [GAIN — quantify benefit] Static cache-token-cost delta model (no live host needed)
- *technique:* Deterministic cost simulator over before/after fixture pairs. For each fixture with a Rule 2 (dynamic content) or Rule 6 (fork-safety) FAIL, build a matched 'fixed' twin (replace $(date) with a frozen ISO string; move the Stop-hook write to a sidecar/SessionStart). Estimate the cached prefix size in tokens (bytes/4, the tool's own TOKEN_ESTIMATE) and model a fixed-length N-turn session under Anthropic's published prompt-cache pricing: cache-write multiplier 1.25x base, cache-read multiplier 0.1x base. A Rule 2/6 FAIL forces a full prefix cache-WRITE every turn (dynamic content/prefix mutation re-keys the cache); the fixed twin pays one write then cache-READs for the remaining N-1 turns. billed_tokens = prefix_tokens * (writes*1.25 + reads*0.1). Report the per-session billed-token delta and percent cache-hit-rate uplift across a 10-fixture set at N=20 turns. This finally produces EVAL.md's flagged-but-unmeasured '>=30% uplift' number, entirely statically.  ·  *determinism:* `deterministic`  ·  *effort:* `med`
- **measures:** Billed prefix-token cost per simulated 20-turn session, before vs after fixing each FAIL, plus the resulting cache-hit-rate uplift in percentage points.
- **gain metric (number+unit + baseline):** Cache-hit-rate uplift in percentage points and billed-prefix-token reduction per 20-turn session (e.g. 'fixing a Rule 2 FAIL on a ~1500-token CLAUDE.md: 0%->95% hit rate, ~26,600 -> ~4,275 billed prefix tokens, an ~84% reduction'), vs the unfixed baseline. Target: median >=30% hit-rate uplift, matching EVAL.md's stated goal.
- **how to run:** New script eval/exp_cache_lint_costmodel/run.py. Reuse exp10's CORPUS entries that carry a FAIL (r2_pos_*, r6_pos_*); for each, programmatically synthesize a 'fixed' twin (regex-replace the dynamic token with a constant; rewrite the hook). Run cache_lint.audit on both to confirm FAIL->OK transition (asserts the fix is real, not assumed). Then compute billed tokens with constants TOKENS=bytes//4, WRITE_MULT=1.25, READ_MULT=0.1, N_TURNS=20: dirty = T*(N*1.25); clean = T*(1.25 + (N-1)*0.1); uplift_pct = (clean_hits/N)*100 where clean_hits=N-1. Emit JSON {fixture, prefix_tokens, billed_dirty, billed_clean, pct_reduction, hitrate_uplift_pp}. Fully deterministic; assert median pct_reduction>=30. Document the pricing multipliers as the explicit baseline so they can be re-pinned if Anthropic changes them.
- **teaches:** Converts cache-lint's boolean FAILs into the dollar/token quantity they actually prevent, and shows that the gain scales with prefix size * turn count — i.e. the rule matters most on large CLAUDE.md files in long sessions, which tells maintainers where to prioritize fixes.
- **improvement surfaced:** Reveals that Rule 5 (sizing) and Rule 2 (dynamic content) interact: a large prefix that ALSO has dynamic content is the worst case, so the tool could rank findings by modeled token cost instead of listing them flat. Also exposes whether the bytes/4 token estimate materially mis-prices small files.

### [STRESS — reliability/efficiency] Property-based fuzz + adversarial perf battery with a hard time budget
- *technique:* Two-part stress test. (A) Property/metamorphic fuzz: generate random Markdown by sampling tokens from {static prose, $(...), ${VAR}, backticked-dynamic, ``` fences, headings, $price}, then assert invariants that must hold regardless of input: (i) audit never raises and always finalizes; (ii) wrapping any matched dynamic token in single backticks never RAISES its severity (typography monotonicity); (iii) moving a token inside a fence never raises FAIL->stays<=WARN; (iv) appending pure static prose never changes existing findings' severities (locality); (v) exit code is a pure function of (#FAIL,#WARN). - **measures:** (A) Number of invariant violations / unhandled exceptions across N>=500 fuzzed inputs (target 0). (B) Wall-clock audit latency in milliseconds at K=1k/5k/20k pathological-token counts, and whether it stays under the asserted budget (i.e. confirms no O(n^2) regression).
- **gain metric (number+unit + baseline):** Invariant-violation count over 500 seeded fuzz cases (target: 0 violations, 0 crashes) and audit latency in ms at K=20000 with a hard ceiling (e.g. assert <1500ms; today ~6ms at K=5000), establishing a perf regression tripwire that does not currently exist.

Part A: 500 generated fixtures; for each, run audit then run a perturbed twin (backtick-wrap / fence-wrap / append-static) and assert the severity-multiset relation holds; collect violations with the offending seed for replay. Part B: time.perf_counter around audit on '$(date) '*K and backticked variants for K in (1000,5000,20000); assert each < its budget; also assert backticked variant (all typography, zero findings) is within ~2x of the bare variant (guards the typography path from quadratic blowup). Deterministic via fixed seed + fixed K; wrap perf asserts with a small tolerance (budget = 6x observed median to absorb CI jitter, variance bound documented).

### [GAIN — quantify benefit] Net-token-economy ROI curve: break-even vs verbosity floor
- *technique:* Extend eval/runner.py into a paired A/B that reports NET token economy, not just output reduction. The existing harness already captures prompt_eval_count (input) and eval_count (output) for both arms; today EVAL.md headlines only delta_output_pct (-86.4%) and separately notes input cost +560%. The gain metric nobody computes is the per-turn NET token delta = (with_in + with_out) - (without_in + without_out), i.e. does the 204-token body + session_start.md pay for itself once the cached body is amortized? Build a fixtures set of prompts stratified by baseline verbosity into 3 buckets: VERBOSE (existing 5 explain/summarize prompts, baseline ~570 out tok), MEDIUM (10 'fix this function / what does this regex do' prompts, baseline ~150 out tok), TERSE-FLOOR (10 imperative prompts 'commit and push','run the tests', baseline <50 out tok matching FINDINGS.md's saturation observation). For each bucket compute mean net-token delta with body-cached (input counted once per session of K=8 turns, amortized) and body-uncached (counted every turn). Emit a break-even table: tokens saved per turn by bucket, and the crossover verbosity (baseline output tokens) below which the skill is net-NEGATIVE.  ·  *determinism:* `seeded-stochastic`  ·  *effort:* `low`
- **measures:** Net tokens saved (or spent) per turn = sum(input+output) with-skill minus without-skill, in whole tokens, broken out by baseline-verbosity bucket and by cached/uncached body accounting; plus the crossover output-token floor where net delta crosses zero.
- **gain metric (number+unit + baseline):** Net tokens saved per turn (whole-token integer, +/-) per verbosity bucket, vs the without-skill arm; e.g. 'VERBOSE: +430 tok/turn saved (cached); TERSE-FLOOR: -180 tok/turn spent (cached) — break-even at ~95 baseline output tokens'. Reported alongside a single headline net-savings number for a representative 8-turn mixed session.
